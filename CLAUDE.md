# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**"As Above, So Below"** is a browser-based hex map tool for generating star systems across four Traveller RPG editions: Classic Traveller (CT), Mongoose Traveller 2nd Edition (MgT2E), Traveller 5 (T5), and Revised Traveller (RTT). It runs as a single HTML file (`hex_map.html`) with vanilla JavaScript ‚Äî no build step, no framework, no bundler.

**To run:** Open `hex_map.html` directly in a browser. There are no build, lint, or test commands.

## Critical Rules (The Sean Protocol)

1. **`rules/` is read-only.** Never edit `rules/ct_data.js`, `rules/mgt2e_data.js`, or `rules/t5_data.js`. These are the authoritative RPG data tables.
2. **Zero-Assumption Policy.** Do not interpret, guess, or "fill in" Traveller RPG rules from training data. If a rule is ambiguous or missing from `rules/`, stop and ask ‚Äî do not implement a best guess.
3. **Halt & Challenge.** When encountering an ambiguity in RPG logic, draft a specific question for the user rather than guessing. Log the issue and stop.
4. **Check `js/` before creating new files.** Prefer extending an existing module over creating a new one.
5. **Do Not Interact with Git.**  All git adding, staging and committing will be handled manually by the user.  You will only work on the code.

## Architecture

The codebase uses a strict 3-layer separation:

| Layer | Location | Role |
|---|---|---|
| Directives (Source of Truth) | `directives/` + `rules/` | SOPs in Markdown; read-only RPG data tables |
| Orchestration | AI agent | Routing, auditing, Halt & Challenge |
| Execution | `js/` | Deterministic JS generation engines |

### Script Load Order (from `hex_map.html`)

Scripts must be loaded in dependency order. The current order is:
1. `js/constants.js` ‚Äî large static data (zone tables, etc.)
2. `js/core.js` ‚Äî global app state, seeded RNG (`mulberry32`/`hashString`), coordinate math
3. `js/renderer.js` ‚Äî canvas drawing
4. Per-edition engines (CT ‚Üí MgT2E ‚Üí T5 ‚Üí RTT), each with `rules/` data loaded first
5. `js/universal_math.js` ‚Äî shared math utilities
6. Input/UI layer: `input_init.js`, `canvas_input.js`, `keyboard_shortcuts.js`, `macro_orchestrator.js`, `hex_editor.js`, `ui_menus.js`, `filter_engine.js`, `io_manager.js`
7. `names.js` (optional ‚Äî generated by `convert_names.ps1`)

### Per-Edition Module Pattern

Each Traveller edition (CT, MgT2E, T5) follows the same module structure:

- `*_world_engine.js` ‚Äî mainworld UWP generation
- `*_stellar_engine.js` ‚Äî star system generation
- `*_socio_engine.js` ‚Äî socioeconomic expansion
- `*_topdown_generator.js` ‚Äî top-down system generation orchestrator
- `*_bottomup_generator.js` ‚Äî bottom-up system generation orchestrator
- `*_uwp_auditor.js` ‚Äî post-generation validation
- `*_system_driver.js` / `system_driver.js` ‚Äî drives full system generation sequence

### Key Global State (in `js/core.js`)

- `hexStates` (Map) ‚Äî all hex data, keyed by hexId
- `masterSeed` + `rng` (mulberry32) ‚Äî deterministic seeded generation; re-seeded per hex via `reseedForHex(hexId)`
- `selectedHexes` (Set) ‚Äî currently selected hexes
- `window.sectorRoutes` ‚Äî trade/xboat route array
- `window.activeFilterRules` ‚Äî filter/color rules

### Trace Logging

Every significant generation step must use the trace logging framework:
- `tSection("description")` ‚Äî open a log section
- `tResult("label", value)` ‚Äî log a result
- `writeLogLine(text)` ‚Äî raw log line

Generation logs are gated on `window.isLoggingEnabled` and downloaded as `.txt` files via `io_manager.js`.

## Version Update Procedure

When bumping the version, update these locations (see `directives/update_version.md`):
1. `js/core.js` ‚Äî `APP_VERSION` and `APP_BANNER` constants
2. `README.md` ‚Äî version header
3. `changelog.md` ‚Äî add entry at top with date and description
4. `hex_map.html` ‚Äî update entries on splash screen and shortcut help screen
